## 4. `src/` 配下の責務

### 4.1 `src/main.ts`

エントリポイント。

- `App.svelte` を mount
- グローバル CSS を読み込み

### 4.2 `src/App.svelte`

画面の最上位コンポーネント。

責務:

- `getSchema()` の呼び出し
- `window.location.pathname` を使ったデータロード
- `popstate` 監視による再ロード
- カテゴリ一覧の保持
- カテゴリ選択状態の保持
- フィルター状態の保持
- ソート状態の保持
- フィルター済みデータの算出
- ソート済みデータの算出
- 子コンポーネントへの受け渡し

ここでは描画の細部は持たず、状態管理中心にしている。

### 4.3 `src/app.css`

アプリ共通 CSS。

主な対象:

- 全体レイアウト
- サイドバー見た目
- テーブル見た目
- ソート列の背景色
- フィルターハイライト
- ソート矢印の CSS 三角形

### 4.4 `src/types.ts`

型定義。

主な型:

- `SchemaColumn`
- `CategorySchema`
- `RecordItem`
- `ViewerData`

### 4.5 `src/schema.json`

カテゴリと列のスキーマ定義。

現状の役割:

- カテゴリ ID
- カテゴリ表示名
- フェッチプログラム名
- 列定義

注意:

- 現時点ではローカル import の静的ファイル
- 日本語文字列は端末表示で文字化けする場合がある

### 4.6 `src/lib/viewerData.ts`

データソースごとの `ViewerData` 組み立て関数を集約したファイル。

現時点の役割:

- 空データ返却
- モックデータ返却
- `/xxx` 用サンプルデータ返却
- `Row` 列の共通付与

関数例:

- `createEmptyViewerData()`
- `createMockViewerData()`
- `createExampleViewerData()`
- `createSchemaWithRowColumn()`

## 7. データフロー

現行の大まかな流れは以下。

1. `App.svelte` が `window.location.pathname` をもとに `getSchema()` を呼ぶ
2. `schema.ts` が URL パスに応じて返却元を切り替える
3. `viewerData.ts` が実際の `ViewerData` を組み立てる
4. `App.svelte` がカテゴリ・フィルター・ソート状態を保持する
5. `App.svelte` がフィルター済み・ソート済みレコードを計算する
6. `TablePanel.svelte` が件数表示とテーブル本体呼び出しを行う
7. `VirtualTable.svelte` が表示範囲のみ仮想描画する

## 10. 保守上の要点

- 状態管理は `App.svelte` に寄せる
- 表示の詳細は子コンポーネントに分ける
- 共通列は `schema.ts` 側で注入する
- URL パスと取得元の対応は `schema.ts` に寄せる
- データ組み立ての実体は `viewerData.ts` に寄せる
- パフォーマンス要件を意識して仮想スクロールを壊さない
- ソート矢印の幅はスキーマ幅と分離する
- フィルターハイライトは HTML エスケープを維持する

この前提を守ると、今後の機能追加でも構造を大きく崩さずに進めやすい。

### 11.9 `src/lib/validation.ts`

後から追加されたランタイム検証モジュール。

役割:

- 初回スキーマロード結果の検証
- カテゴリレコード取得結果の検証

主な関数:

- `validateViewerData()`
- `validateRecords()`

エラー時の扱い:

- スキーマ検証失敗
  - `App.svelte` のロードエラー表示へ流す
- レコード検証失敗
  - `App.svelte` のカテゴリロードエラー表示へ流す

外部フェッチを本実装化する場合、このファイルのルールも合わせて見直すこと

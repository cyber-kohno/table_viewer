## 4. `src/` 配下の責務

### 4.1 `src/main.ts`

エントリポイント。

- `App.svelte` を mount
- グローバル CSS を読み込み

### 4.2 `src/App.svelte`

画面の最上位コンポーネント。

責務:

- `getSchema()` の呼び出し
- `window.location.pathname` を使ったデータロード
- `popstate` 監視による再ロード
- カテゴリ一覧の保持
- カテゴリ選択状態の保持
- フィルター状態の保持
- ソート状態の保持
- フィルター済みデータの算出
- ソート済みデータの算出
- 子コンポーネントへの受け渡し

ここでは描画の細部は持たず、状態管理中心にしている。

### 4.3 `src/app.css`

アプリ共通 CSS。

主な対象:

- 全体レイアウト
- サイドバー見た目
- テーブル見た目
- ソート列の背景色
- フィルターハイライト
- ソート矢印の CSS 三角形

### 4.4 `src/types.ts`

型定義。

主な型:

- `SchemaColumn`
- `CategorySchema`
- `RecordItem`
- `ViewerData`

### 4.5 `src/schema.json`

カテゴリと列のスキーマ定義。

現状の役割:

- カテゴリ ID
- カテゴリ表示名
- フェッチプログラム名
- 列定義

注意:

- 現時点ではローカル import の静的ファイル
- 日本語文字列は端末表示で文字化けする場合がある

### 4.6 `src/lib/viewerData.ts`

データソースごとの `ViewerData` 組み立て関数を集約したファイル。

現時点の役割:

- 空データ返却
- モックデータ返却
- `/xxx` 用サンプルデータ返却
- `Row` 列の共通付与

関数例:

- `createEmptyViewerData()`
- `createMockViewerData()`
- `createExampleViewerData()`
- `createSchemaWithRowColumn()`

## 5. `src/components/` 配下

### 5.1 `src/components/CategorySidebar.svelte`

左側サイドバー表示専用コンポーネント。

責務:

- カテゴリ一覧の描画
- 選択中カテゴリの見た目切り替え
- クリック時のカテゴリ選択通知

持たないもの:

- カテゴリ状態そのもの
- データ取得処理

### 5.2 `src/components/TablePanel.svelte`

右側パネルの枠コンポーネント。

責務:

- 未選択時の空パネル表示
- 件数表示
- `VirtualTable.svelte` の呼び出し

### 5.3 `src/components/VirtualTable.svelte`

テーブル描画の中核コンポーネント。

責務:

- ヘッダ描画
- フィルター行描画
- ソート矢印表示
- ソート列背景色付け
- フィルター一致文字列ハイライト
- 仮想スクロール
- 縦スクロール位置に応じた表示行切り出し
- 横スクロール可能なテーブル描画

重要な内部ロジック:

- `sortIndicatorWidth`
  - ソート矢印ぶんの追加幅
- `getSortState()`
  - ソート状態の判定
- `getHighlightedCell()`
  - フィルター一致文字列を `mark` 付き HTML に変換
- `visibleCount / startIndex / endIndex`
  - 仮想スクロール計算

変更時の注意:

- スクロール不具合が起きやすい箇所
- 列幅・固定ヘッダ・仮想スクロールが密結合している
- 変更後は必ずビルド・表示確認が必要

## 6. `src/lib/` 配下

### 6.1 `src/lib/schema.ts`

スキーマ取得の窓口。

責務:

- URL パスの正規化
- URL パスに応じたデータソース切り替え
- `ViewerData` の返却

現在の分岐:

- `/`
  - 空データ
- `/mock`
  - モックデータ
- `/xxx`
  - 任意実装のサンプルデータ
- その他
  - 空データ

将来的に置き換わる可能性が高い箇所:

- API 呼び出し
- SQLite 連携
- `fetchProgram` による取得分岐
- URL パスごとの本実装追加

### 6.2 `src/lib/mockData.ts`

モックレコード生成。

現在ある関数:

- `createMockTableRecords`
- `createMockColumnRecords`
- `createMockMessageRecords`

役割:

- 開発中に UI 確認できるだけのダミーデータを返す

## 7. データフロー

現行の大まかな流れは以下。

1. `App.svelte` が `window.location.pathname` をもとに `getSchema()` を呼ぶ
2. `schema.ts` が URL パスに応じて返却元を切り替える
3. `viewerData.ts` が実際の `ViewerData` を組み立てる
4. `App.svelte` がカテゴリ・フィルター・ソート状態を保持する
5. `App.svelte` がフィルター済み・ソート済みレコードを計算する
6. `TablePanel.svelte` が件数表示とテーブル本体呼び出しを行う
7. `VirtualTable.svelte` が表示範囲のみ仮想描画する

## 8. 今後の拡張で主に触る場所

### 8.1 新カテゴリを追加したい場合

主に触る場所:

- `src/schema.json`
- 必要なら `src/lib/mockData.ts`
- 必要なら `src/lib/schema.ts`

### 8.2 SQLite 連携したい場合

主に触る場所:

- `src/lib/schema.ts`
- `src/lib/viewerData.ts`
- 必要に応じて新しいデータ取得モジュール追加

### 8.2.1 URL パスで取得元を増やしたい場合

主に触る場所:

- `src/lib/schema.ts`
- `src/lib/viewerData.ts`

典型例:

- `/sqlite`
- `/api`
- `/staging`
- `/customer-a`

### 8.3 キーボード操作や選択状態を追加したい場合

主に触る場所:

- `src/App.svelte`
- `src/components/VirtualTable.svelte`

### 8.4 見た目や配色を変えたい場合

主に触る場所:

- `src/app.css`

### 8.5 スキーマ表現を増やしたい場合

主に触る場所:

- `src/types.ts`
- `src/schema.json`
- `src/lib/schema.ts`
- `src/components/VirtualTable.svelte`

## 9. 開発時の確認コマンド

`app` ディレクトリで実行する。

```powershell
npm run build
npm run check
```

確認内容:

- `build`
  - Vite ビルドが通るか
- `check`
  - Svelte / TypeScript 診断に問題がないか

---

## 10. 保守上の要点

- 状態管理は `App.svelte` に寄せる
- 表示の詳細は子コンポーネントに分ける
- 共通列は `schema.ts` 側で注入する
- URL パスと取得元の対応は `schema.ts` に寄せる
- データ組み立ての実体は `viewerData.ts` に寄せる
- パフォーマンス要件を意識して仮想スクロールを壊さない
- ソート矢印の幅はスキーマ幅と分離する
- フィルターハイライトは HTML エスケープを維持する

この前提を守ると、今後の機能追加でも構造を大きく崩さずに進めやすい。

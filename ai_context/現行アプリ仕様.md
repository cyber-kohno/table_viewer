# 現行アプリ仕様

## 1. このドキュメントの目的

本ドキュメントは、`table_viewer` プロジェクトの現行フロントエンド実装について、次の AI エージェントまたは開発者がコンテキストなしでも継続開発できるようにするための引き継ぎ資料である。

- 何を作っているアプリか
- 現時点で何が実装済みか
- どのような方針で実装しているか
- 今後どこを拡張すべきか

詳細なファイル配置は `ai_context/実装マップ.md` を参照。

---

## 2. アプリの目的

本アプリは、複数の定義書から抽出した定義情報を、カテゴリごとに一覧表示・検索・参照するための汎用ビューアである。

元々の構想では、以下を目指している。

- 定義データ本体は SQLite で管理する
- 表示内容はスキーマ JSON によって外部定義する
- ビューア本体は「表示責務」に専念する
- 項目追加時に、ビューアのハードコード修正を最小化する

現時点では、**SQLite 連携は未実装**であり、**モックスキーマ + モックデータで動作する汎用ビューアの土台**ができている状態である。

## 3. 現時点で実装済みの機能概要

### 3.1 画面レイアウト

- 左側にカテゴリ一覧のサイドバーを表示
- 右側にテーブル表示エリアを表示
- サイドバー幅は約 200px
- モバイル幅では縦積みに近い表示へ崩れるレスポンシブ対応あり

### 3.2 カテゴリ切り替え

- カテゴリをクリックすると右側のテーブル内容が切り替わる
- 初期状態ではカテゴリ未選択
- 選択中カテゴリをもう一度クリックすると未選択状態へ戻る
- 未選択時は右側テーブルエリアは空表示

### 3.3 スキーマ駆動のテーブル表示

- 表示列は `src/schema.json` の定義から生成する
- 各カテゴリは少なくとも以下を持つ
  - `id`
  - `categoryName`
  - `fetchProgram`
  - `columns[]`
- `columns[]` の各要素は以下を持つ
  - `label`
  - `key`
  - `width`

### 3.4 モックカテゴリ

現時点では以下のダミーカテゴリを実装している。

- テーブル
- カラム
- メッセージ

### 3.4.1 URL パスによるデータ切り替え

現時点では、取得するスキーマ・データは URL パスによって切り替わる。

- `/`
  - 何も取得しない
  - エラーにはせず、空データを返す
  - カテゴリもテーブルも表示しない
- `/mock`
  - 現在のモック実装を返す
  - テーブル、カラム、メッセージのダミーカテゴリを表示する
- `/xxx`
  - 任意実装のサンプル経路
  - 現在は `Example` カテゴリを返すダミー実装になっている
- 上記以外のパス
  - エラーにはせず、空データを返す

この設計により、今後 `/yyy` などを追加して取得元を段階的に差し替えられる。

### 3.5 Row 列

- 全カテゴリに共通で `Row` 列を先頭に自動付与している
- `Row` 列は実データではなく表示専用列
- フィルター対象外
- ソート対象外
- フィルター後・ソート後の表示順に応じて 1 から採番される

### 3.6 フィルター

- 各列ごとに部分一致フィルターが可能
- フィルターは大文字小文字を区別しない
- `Row` 列のみフィルター入力欄を表示しない
- フィルター中は、該当列の一致文字列をハイライト表示する

### 3.7 ソート

- ヘッダクリックで昇順・降順ソート可能
- 対象列を初めてクリックすると昇順
- 同じ列を再クリックすると降順
- `Row` 列はソート対象外
- ソート中の列は以下で視覚化される
  - ヘッダの矢印
  - 列全体の薄い背景色

### 3.8 件数表示

- テーブル上部に `フィルター後件数 / 元件数` を表示
- 文言自体は固定ラベルではなく、数字と `/` のみ表示

### 3.9 仮想スクロール

- レコード数が多い前提のため、全件 DOM 描画はしない
- 画面上に見えている範囲 + オーバースキャン分のみ描画
- 縦スクロール時に可視範囲のみ差し替える
- ヘッダ行・フィルター行は固定表示
- 横方向はテーブル領域内でスクロール

### 3.10 表示言語

- モックデータ以外のアプリ固定文言は英語
- ただし引き継ぎドキュメントは日本語

## 5. 現行の実装方針

### 5.1 ビューア責務に寄せる

アプリ本体は、できるだけ以下だけを担当する方針。

- カテゴリ表示
- テーブル表示
- フィルター
- ソート
- 仮想スクロール
- スキーマに基づく動的 UI 構成
- URL パスに応じた取得元切り替え

データの意味やカテゴリ固有の業務ロジックは、将来的にスキーマまたは取得層に寄せる想定。

### 5.2 親は状態管理、子は表示責務

`App.svelte` は以下だけに絞っている。

- 初期ロード
- 選択カテゴリ状態
- フィルター状態
- ソート状態
- フィルター済み / ソート済みレコードの算出

表示の詳細は子コンポーネントに分離している。

### 5.3 スキーマで列を定義し、共通列はコードで補う

- 各カテゴリ固有列は `schema.json` で持つ
- `Row` 列のような共通制御列は `schema.ts` 側で先頭追加する

この方針により、カテゴリ定義の最小記述を維持しつつ、ビューア共通機能を一箇所で管理できる。

### 5.3.1 取得元分岐は `getSchema()` に集約する

URL パスごとの切り替えは `getSchema(pathname)` に集約している。

- 画面側は「今のパスでロードする」ことだけ意識する
- パスと取得元の対応関係は取得層で完結させる
- 将来の `/sqlite` `/api` 追加時も UI 側の変更を最小にする

### 5.4 パフォーマンス重視

- 1000 件以上を想定して仮想スクロールを採用
- 表示中の DOM 数を抑える
- 固定ヘッダ + 可視行のみ描画

### 5.5 Tailwind は使わず純粋な CSS

初期要件通り、列幅の動的制御をしやすくするため Tailwind は不使用。

- グローバル CSS は `src/app.css`
- コンポーネント側はクラス指定中心

### 5.6 ソート矢印の幅はスキーマ幅と分離

要件により、ソート矢印の表示で列幅が圧迫されないようにしている。

- スキーマの `width` は本来の列表示幅
- ソート矢印分の 20px は別枠で加算

## 8. 次のエージェントが理解しておくべき注意点

### 8.1 現在はモックアプリである

本番連携はまだ入っていない。現状は「汎用ビューアの土台づくり」が主目的。

### 8.2 `schema.json` の日本語は文字化けして見える場合がある

PowerShell の表示や文字コードの扱いによっては、日本語文字列が端末上で文字化けして見えることがある。

ただし、実ファイル自体は編集内容を確認して扱うこと。端末出力だけで破損判断しないこと。

### 8.3 URL パス変更時の再ロード

`App.svelte` では `window.location.pathname` をもとに `getSchema()` を呼んでいる。

また、`popstate` を監視して再ロードするため、ブラウザの戻る・進むによるパス変化にも追従する。

ただし現時点では、アプリ内に URL を切り替える UI は持っていない。

### 8.4 `VirtualTable.svelte` は UI の中核

以下の変更は特に影響範囲が大きい。

- 仮想スクロール方式の変更
- ヘッダ構造の変更
- セル描画方法の変更
- フィルターハイライトの変更

ここを触る際は、スクロール不具合と幅崩れに注意すること。

### 8.5 フィルターとソートは親コンポーネントで状態管理

テーブル表示だけを見てロジックを追加しようとすると、状態の責務が崩れる可能性がある。

- 状態計算は `App.svelte`
- 描画は `TablePanel.svelte` / `VirtualTable.svelte`

の分離を維持するのが望ましい。

## 11. 追補: 追加実装済み機能

このセクションは、初回作成時の仕様書に対して後から追加された実装内容を、次の担当者向けに補足するものである。

### 11.1 カテゴリ選択時の遅延ロード

- `getSchema()` はカテゴリ定義だけを返す
- 各カテゴリは `fetchRecords(): Promise<RecordItem[]>` を持つ
- カテゴリをクリックした時に、そのカテゴリの `fetchRecords()` を実行してレコードを遅延取得する
- 一度取得したカテゴリレコードはキャッシュし、同一カテゴリの再表示時は再取得しない
- モック実装では、DB アクセス時間を模擬するために 500ms の待機を入れている

### 11.2 `ViewerData` の考え方

- 旧来の `schema + recordsByCategory` 形式は廃止済み
- 現在の `ViewerData` は `CategorySchema[]` のみ
- レコード群はスキーマ内に持たず、`fetchRecords` のコールバック経由で取得する

### 11.3 関連遷移

`CategorySchema` には `relations: CategoryRelation[]` を持たせている。

各 `CategoryRelation` は以下を持つ。

- 関連定義名
- 遷移先カテゴリ ID
- マッピング定義配列

マッピング定義は、遷移元キーと遷移先キーの組を複数持てる。

- 遷移元レコードの値
- 遷移先カテゴリの対象カラム

を完全一致条件として扱い、**固定フィルター**として内部適用する。

この固定フィルターは、ヘッダの自由入力フィルターとは別枠で扱う。

### 11.4 テーブル⇔カラムのモック相互遷移

現時点ではモックデータで以下が実装済み。

- `tables` から `columns` への遷移
- `columns` から `tables` への遷移

どちらも `tableName` を完全一致条件として関連付けている。

### 11.5 レコード選択

- レコードはクリックで選択できる
- 選択済みレコードを再クリックすると選択解除
- 選択状態は黄色背景で表示
- ソート列と選択行が交差するセルは別色で表示し、状態が両方分かるようにしている

### 11.6 フッタの関連操作

テーブルエリア下部に 1 行分のフッタを確保している。

フッタには以下を表示する。

- `Back` ボタン
- 現在カテゴリが持つ関連定義ボタン群

挙動:

- レコード未選択時は関連定義ボタンは非活性
- レコード選択時のみ関連定義ボタン活性
- `Back` は関連遷移履歴がある時だけ活性

### 11.7 戻る動作と履歴管理

関連遷移時は、以下を履歴として保存している。

- 遷移前カテゴリ ID
- 自由フィルター状態
- ソート状態
- 選択レコード
- スクロール位置
- 固定フィルター状態

`Back` 時は上記を復元する。

さらに、関連先から更に関連遷移できるように、履歴はスタック形式で複数段保持している。

### 11.8 サイドバーの関連遷移表示

- 通常のカテゴリ選択時は従来色
- 関連遷移中の遷移先カテゴリは赤背景で表示

### 11.9 遷移履歴表示

件数表示の右側に履歴表示エリアを持つ。

表示ルール:

- 履歴がない場合でも、選択中カテゴリだけ表示
- 関連遷移時は `>` 区切りで表示
- 遷移先にはマッピング値を `[value1, value2]` 形式で表示
- カテゴリ名とマッピング値は色分けしている
- 最新要素だけ青文字 + 太字
- 過去履歴カテゴリは黒系
- マッピング値は薄い灰色
- 履歴エリア全体は白に近い薄い水色背景

例:

- `テーブル`
- `テーブル > カラム[T_会計_0002]`

### 11.10 件数表示の定義

件数表示は以下の意味で表示している。

- 分母: 関連固定フィルター適用後の件数
- 分子: さらに自由フィルター適用後の件数

つまり、関連遷移中は「元データ全件」ではなく、「関連条件で絞られた集合」を母数とする。

### 11.11 キーボード操作

現時点で以下が利用可能。

- `↑`
  - フォーカスレコードを 1 行上へ移動
- `↓`
  - フォーカスレコードを 1 行下へ移動
- `→`
  - 関連定義が 1 件以上ある場合、0 番目の関連へ遷移
- `←`
  - `Back` 実行

備考:

- フィルター入力中はキー操作を奪わない
- 上下移動時はフォーカス中レコードが中央付近に来るようスクロール位置を再計算する

### 11.12 スクロール復元

関連遷移前のスクロール位置は履歴に保存している。

`Back` 時は、

- カテゴリ
- 選択レコード
- フィルター
- ソート
- スクロール位置

をまとめて復元する。

### 11.13 コピー機能

#### 11.13.1 表示中レコード一括コピー

- `Row` 列のフィルター行には `Copy` ボタンを表示
- 押下時、現在表示中のレコードを TSV 形式でクリップボードへコピー
- ヘッダ行も含める
- コピー対象は、自由フィルター・関連固定フィルター・現在の並び順を反映した結果

#### 11.13.2 セル単体コピー

- セルを右クリックすると、そのセル値をクリップボードへコピー
- 左クリックはレコード選択のまま

### 11.14 トースト通知

- コピー完了時にはトーストを表示
- トーストはピンク系
- マウス位置の近くに表示
- フェードアウトしながら消える
- 表示時間は約 1.5 秒

### 11.15 ランタイムバリデーション

外部フェッチ前提のため、TypeScript の型だけではなくランタイムでも検証する。

#### 11.15.1 初回スキーマロード時

- `getSchema()` の戻り値に対してスキーマ構造を検証する
- 想定と異なる構造であれば、画面全体のロードエラーとして表示する

検証対象の例:

- 配列であること
- 各カテゴリが `id` / `categoryName` / `fetchRecords` / `columns` / `relations` を持つこと
- 各カラムが `label` / `key` / `width` を持つこと
- 各 relation / mapping が必要プロパティを持つこと

#### 11.15.2 カテゴリレコード取得時

- カテゴリクリック時の `fetchRecords()` 戻り値に対してレコード構造を検証する
- 想定と異なる構造であれば、そのカテゴリエリアにエラー表示する

検証対象の例:

- 配列であること
- 各レコードが object であること
- スキーマで定義された列キーを全て持つこと
- 値が `string` または `number` であること

これにより、外部 API / DB から壊れたデータが返っても、最低限どこで壊れているか分かる状態にしている

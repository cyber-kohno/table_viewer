# 現行アプリ仕様

## 1. このドキュメントの目的

本ドキュメントは、`table_viewer` プロジェクトの現行フロントエンド実装について、次の AI エージェントまたは開発者がコンテキストなしでも継続開発できるようにするための引き継ぎ資料である。

- 何を作っているアプリか
- 現時点で何が実装済みか
- どのような方針で実装しているか
- 今後どこを拡張すべきか

詳細なファイル配置は `ai_context/実装マップ.md` を参照。

---

## 2. アプリの目的

本アプリは、複数の定義書から抽出した定義情報を、カテゴリごとに一覧表示・検索・参照するための汎用ビューアである。

元々の構想では、以下を目指している。

- 定義データ本体は SQLite で管理する
- 表示内容はスキーマ JSON によって外部定義する
- ビューア本体は「表示責務」に専念する
- 項目追加時に、ビューアのハードコード修正を最小化する

現時点では、**SQLite 連携は未実装**であり、**モックスキーマ + モックデータで動作する汎用ビューアの土台**ができている状態である。

## 3. 現時点で実装済みの機能概要

### 3.1 画面レイアウト

- 左側にカテゴリ一覧のサイドバーを表示
- 右側にテーブル表示エリアを表示
- サイドバー幅は約 200px
- モバイル幅では縦積みに近い表示へ崩れるレスポンシブ対応あり

### 3.2 カテゴリ切り替え

- カテゴリをクリックすると右側のテーブル内容が切り替わる
- 初期状態ではカテゴリ未選択
- 選択中カテゴリをもう一度クリックすると未選択状態へ戻る
- 未選択時は右側テーブルエリアは空表示

### 3.3 スキーマ駆動のテーブル表示

- 表示列は `src/schema.json` の定義から生成する
- 各カテゴリは少なくとも以下を持つ
  - `id`
  - `categoryName`
  - `fetchProgram`
  - `columns[]`
- `columns[]` の各要素は以下を持つ
  - `label`
  - `key`
  - `width`

### 3.4 モックカテゴリ

現時点では以下のダミーカテゴリを実装している。

- テーブル
- カラム
- メッセージ

### 3.4.1 URL パスによるデータ切り替え

現時点では、取得するスキーマ・データは URL パスによって切り替わる。

- `/`
  - 何も取得しない
  - エラーにはせず、空データを返す
  - カテゴリもテーブルも表示しない
- `/mock`
  - 現在のモック実装を返す
  - テーブル、カラム、メッセージのダミーカテゴリを表示する
- `/xxx`
  - 任意実装のサンプル経路
  - 現在は `Example` カテゴリを返すダミー実装になっている
- 上記以外のパス
  - エラーにはせず、空データを返す

この設計により、今後 `/yyy` などを追加して取得元を段階的に差し替えられる。

### 3.5 Row 列

- 全カテゴリに共通で `Row` 列を先頭に自動付与している
- `Row` 列は実データではなく表示専用列
- フィルター対象外
- ソート対象外
- フィルター後・ソート後の表示順に応じて 1 から採番される

### 3.6 フィルター

- 各列ごとに部分一致フィルターが可能
- フィルターは大文字小文字を区別しない
- `Row` 列のみフィルター入力欄を表示しない
- フィルター中は、該当列の一致文字列をハイライト表示する

### 3.7 ソート

- ヘッダクリックで昇順・降順ソート可能
- 対象列を初めてクリックすると昇順
- 同じ列を再クリックすると降順
- `Row` 列はソート対象外
- ソート中の列は以下で視覚化される
  - ヘッダの矢印
  - 列全体の薄い背景色

### 3.8 件数表示

- テーブル上部に `フィルター後件数 / 元件数` を表示
- 文言自体は固定ラベルではなく、数字と `/` のみ表示

### 3.9 仮想スクロール

- レコード数が多い前提のため、全件 DOM 描画はしない
- 画面上に見えている範囲 + オーバースキャン分のみ描画
- 縦スクロール時に可視範囲のみ差し替える
- ヘッダ行・フィルター行は固定表示
- 横方向はテーブル領域内でスクロール

### 3.10 表示言語

- モックデータ以外のアプリ固定文言は英語
- ただし引き継ぎドキュメントは日本語

## 4. 実装済みだが未完成・今後拡張予定の領域

未実装。現在は `getSchema()` がモックデータを返しているだけ。

将来的には以下を想定。

- SQLite からカテゴリごとのデータ取得
- `fetchProgram` をもとに取得処理を切り替える仕組み
- リソース単位の部分更新

### 4.2 スキーマの外部取得

未実装。現在は `src/schema.json` をローカル import している。

将来的には以下を想定。

- API または外部 JSON ファイル取得
- スキーマ更新だけで画面表示列を変更できる状態
- URL パスごとに異なる取得先を切り替える状態

### 4.3 関連データ遷移

元要件にあった以下は未実装。

- テーブル行を選択して右キーでカラム一覧へ遷移
- 左キーで遷移元へ戻る
- 行選択状態を前提としたキーボードナビゲーション

### 4.4 フェッチプログラムの意味づけ

`fetchProgram` フィールドは現時点では表示定義上のダミー値であり、実処理には使用していない。

### 4.5 URL パスごとの本実装差し替え

`/mock` と `/xxx` は、取得元差し替えを意識した構造になっている。

今後の想定例:

- `/mock`
  - 開発確認用のモック固定
- `/local`
  - ローカルファイル読み込み
- `/sqlite`
  - SQLite ベース実装
- `/api`
  - API ベース実装

現時点では `/xxx` が「任意の実装を差し込む例」として存在している。

## 5. 現行の実装方針

### 5.1 ビューア責務に寄せる

アプリ本体は、できるだけ以下だけを担当する方針。

- カテゴリ表示
- テーブル表示
- フィルター
- ソート
- 仮想スクロール
- スキーマに基づく動的 UI 構成
- URL パスに応じた取得元切り替え

データの意味やカテゴリ固有の業務ロジックは、将来的にスキーマまたは取得層に寄せる想定。

### 5.2 親は状態管理、子は表示責務

`App.svelte` は以下だけに絞っている。

- 初期ロード
- 選択カテゴリ状態
- フィルター状態
- ソート状態
- フィルター済み / ソート済みレコードの算出

表示の詳細は子コンポーネントに分離している。

### 5.3 スキーマで列を定義し、共通列はコードで補う

- 各カテゴリ固有列は `schema.json` で持つ
- `Row` 列のような共通制御列は `schema.ts` 側で先頭追加する

この方針により、カテゴリ定義の最小記述を維持しつつ、ビューア共通機能を一箇所で管理できる。

### 5.3.1 取得元分岐は `getSchema()` に集約する

URL パスごとの切り替えは `getSchema(pathname)` に集約している。

- 画面側は「今のパスでロードする」ことだけ意識する
- パスと取得元の対応関係は取得層で完結させる
- 将来の `/sqlite` `/api` 追加時も UI 側の変更を最小にする

### 5.4 パフォーマンス重視

- 1000 件以上を想定して仮想スクロールを採用
- 表示中の DOM 数を抑える
- 固定ヘッダ + 可視行のみ描画

### 5.5 Tailwind は使わず純粋な CSS

初期要件通り、列幅の動的制御をしやすくするため Tailwind は不使用。

- グローバル CSS は `src/app.css`
- コンポーネント側はクラス指定中心

### 5.6 ソート矢印の幅はスキーマ幅と分離

要件により、ソート矢印の表示で列幅が圧迫されないようにしている。

- スキーマの `width` は本来の列表示幅
- ソート矢印分の 20px は別枠で加算

## 6. 画面仕様の詳細

### 6.1 サイドバー

- タイトル: `Categories`
- カテゴリボタン押下で選択切り替え
- 選択中カテゴリは色反転で表示

### 6.2 テーブルヘッダ

- 列名を表示
- ソート対象列のみクリック可能
- ソート中列には CSS 三角形で昇順・降順矢印を表示
- ソート中列は背景色を薄く色付け

### 6.3 フィルター行

- 各列ごとに入力欄を表示
- `Row` 列のみ入力欄なし
- ソート中列はフィルターセルも薄く色付け

### 6.4 ボディ

- 仮想スクロールで表示範囲のみ描画
- 偶数・奇数行で背景を切り替え
- ホバー時に行背景を変更
- ソート中列はセル背景も薄く色付け
- フィルター一致箇所は黄色系の `mark` で強調

## 8. 次のエージェントが理解しておくべき注意点

### 8.1 現在はモックアプリである

本番連携はまだ入っていない。現状は「汎用ビューアの土台づくり」が主目的。

### 8.2 `schema.json` の日本語は文字化けして見える場合がある

PowerShell の表示や文字コードの扱いによっては、日本語文字列が端末上で文字化けして見えることがある。

ただし、実ファイル自体は編集内容を確認して扱うこと。端末出力だけで破損判断しないこと。

### 8.3 URL パス変更時の再ロード

`App.svelte` では `window.location.pathname` をもとに `getSchema()` を呼んでいる。

また、`popstate` を監視して再ロードするため、ブラウザの戻る・進むによるパス変化にも追従する。

ただし現時点では、アプリ内に URL を切り替える UI は持っていない。

### 8.4 `VirtualTable.svelte` は UI の中核

以下の変更は特に影響範囲が大きい。

- 仮想スクロール方式の変更
- ヘッダ構造の変更
- セル描画方法の変更
- フィルターハイライトの変更

ここを触る際は、スクロール不具合と幅崩れに注意すること。

### 8.5 フィルターとソートは親コンポーネントで状態管理

テーブル表示だけを見てロジックを追加しようとすると、状態の責務が崩れる可能性がある。

- 状態計算は `App.svelte`
- 描画は `TablePanel.svelte` / `VirtualTable.svelte`

の分離を維持するのが望ましい。

## 9. 今後の推奨拡張順

次の順で進めるのが安全。

1. `getSchema()` の返却形式を維持したまま、モック取得層を API / SQLite 取得層へ差し替える
2. `/xxx` を本物の任意実装へ置き換える、または `/sqlite` `/api` などを追加する
3. `fetchProgram` の解釈ルールを定義する
4. 行選択状態とキーボードナビゲーションを追加する
5. 関連カテゴリ遷移を追加する
6. スキーマでソート可否・フィルター可否・ハイライト可否なども制御できるようにする

## 10. 現在の到達点まとめ

現時点のアプリは、以下を満たしている。

- スキーマ駆動でカテゴリ・列を表示できる
- カテゴリ切り替えができる
- 初期未選択状態を持てる
- 部分一致フィルターができる
- ソートができる
- フィルター一致箇所をハイライトできる
- ソート列を視覚強調できる
- 大量件数でも仮想スクロールで扱える
- コンポーネント分割済みで保守しやすい

一方で、以下は今後の実装対象である。

- SQLite 連携
- 本物のフェッチ処理
- URL パスごとの実運用データソース追加
- キーボード操作
- 関連データ遷移
- より汎用的なスキーマ制御
